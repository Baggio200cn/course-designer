# 必须停下来问用户的触发条件

> **遇到以下任一情况，Claude 必须暂停并向用户说明，不能自行决定。**

| # | 触发条件 | 应对方式 |
|---|---------|---------|
| 1 | 改动涉及 `contracts.js` / `publication-contracts.js` | 停下来说明影响（关联 H1）|
| 2 | 需要新增一个 AI API 调用 | 说明用途、token 估算、fallback 方案 |
| 3 | 重构范围超过 2 个文件 | 说明为什么需要跨文件改动（关联 H6）|
| 4 | 有现有测试脚本会 FAIL | 不要修改脚本去适配，先告知用户（关联 H4）|
| 5 | 要引入新的 npm 包 | 说明理由和替代方案（关联 H8）|
| 6 | 发现 quality.js 或 abc-generator.js 的逻辑疑似有 bug | 说明，不要自己改 |
| 7 | 同一个 prompt 逻辑在多处地方存在 | 说明，等迁移到 Registry 再统一（关联 H5）|
| 8 | Stage 状态机有任何不确定的地方 | 说明，不要猜测 |
| 9 | Agent 决策逻辑修改可能影响已有 Stage 流程 | 说明影响范围，不要自行判断 |
| 10 | 想新建一个 .md 放在项目根目录 | 先查 `.claude/file-placement-guide.md`，正确归位 |
| 11 | 想升级一条新硬约束（H14+）| 先放 `.claude/candidates/`，重复 3 次再升级 |
| 12 | 用户口头说"顺便改一下 X" | 确认是否在当前任务范围内（关联 H6）|

## 暂停时的标准格式

```
我准备做 X，但触发了停止条件 N（"为什么停止"原因）：

具体影响：
- ...
- ...

可选方案：
A. ...
B. ...

请确认走哪条，或者你有其他想法？
```

## 停下来的好处

- 避免"自以为对"的代码改动后被推翻
- 留给用户审视空间，符合 0.3 节"报告时禁用总结性大词"
- 让用户感知到 Claude 在思考边界，而非盲目执行

## 反例（不要这样做）

- ❌ "我顺便把 contracts.js 改了一下，应该没问题"
- ❌ "我安装了 lodash，因为代码会更简洁"（没问就装 = 违反 H8）
- ❌ "我新建了 TODO.md 在根目录"（没查 file-placement-guide）
- ❌ "我重构了 5 个文件因为它们都用了相同模式"（违反 H6）
