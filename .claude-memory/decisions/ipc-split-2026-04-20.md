## 确认的方案

12 组文件，v2-stage 按阶段拆 4 个子文件：

```
src/main/ipc/
├── _registry.js
├── notebook.handlers.js       (5)
├── module.handlers.js         (7)
├── framework.handlers.js      (8)
├── course.handlers.js         (6)
├── lecture.handlers.js        (4)
├── v2/
│   ├── framework.handlers.js  (framework stage handlers)
│   ├── lecture.handlers.js    (lecture stage handlers)
│   ├── ppt.handlers.js        (ppt stage handlers)
│   └── video.handlers.js      (video stage + workflow/artifact/event)
├── export.handlers.js         (8)
├── resource.handlers.js       (11)
├── media.handlers.js          (6)
├── prompt.handlers.js         (8)
└── system.handlers.js         (9)
```

## 为什么 v2-stage 按阶段拆（不按功能类型拆）

备选方案一：按 CRUD/AI调用/导出 功能类型拆  
备选方案二：按业务 stage 拆（选择此方案）

选择 stage 拆分的理由：
1. contracts.js 已经定义了 stage 是最高层业务抽象
2. 每个 stage 的 handler 有紧密的业务关联（save→confirm→generate 一套）
3. 后续调试某个 stage 问题时，只需要打开对应 stage 文件
4. Claude Code 每次只看一个 stage 文件，上下文更聚焦

## 迁移原则

- 每次只迁移一个分组
- 迁移后必须 `npm run dev` 验证启动正常
- 不能同时修改 handler 里的业务逻辑（先迁移，后优化）
- index.js 里被迁走的部分改为 `require('./ipc/xxx.handlers').register(ipcMain)`
